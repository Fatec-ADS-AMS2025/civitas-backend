## Security
- JWT authentication is configured but not enforced on the API surface. `Civitas.WebAPI/Program.cs` registers `AddAuthentication()`/`AddAuthorization()`, but the only controller attribute I found is `[AllowAnonymous]` on `Civitas.WebAPI/Controllers/AuthController.cs`. No other controller in `Civitas.WebAPI/Controllers/` is protected with `[Authorize]`, so the bearer-token setup is effectively bypassed.
- Sensitive defaults are committed to source control. `Civitas.WebAPI/appsettings.json` contains a real PostgreSQL password (`Password=123456`) and a JWT signing secret (`development-only-key-change-before-production-2026`). That makes the repo itself a secret distribution point and creates a high risk of environment reuse or accidental production deployment with weak credentials.
- Error handling leaks internal details to clients. Several controllers return `ex.Message` and `StackTrace` in 500 responses, including `Civitas.WebAPI/controllers/FornecedorController.cs`, `Civitas.WebAPI/Controllers/DespesaController.cs`, `Civitas.WebAPI/Controllers/AuthController.cs`, and similar patterns in `Civitas.WebAPI/Controllers/SecretariaController.cs` and `Civitas.WebAPI/Controllers/FluxoController.cs`. This is information disclosure and also makes the API contract unstable.

## Documentation Drift
- Runtime ports in the docs do not match the actual launch profile. `README.md` says Swagger runs on `http://localhost:5000` and `https://localhost:5001`, but `Civitas.WebAPI/Properties/launchSettings.json` is configured for `http://localhost:5210` and `https://localhost:7058`. The `.http` scratch file follows the launch settings, so the README is stale.
- Endpoint examples and comments are inconsistent with the actual route naming. `Civitas.WebAPI/controllers/FornecedorController.cs` has comments for `api/Fornecedor` while the route is `api/fornecedores`, and `Civitas.WebAPI/Controllers/DespesaController.cs` repeats the same singular/plural mismatch in its examples. This is minor by itself, but it increases the chance of copying incorrect URLs into clients and tests.
- The README implies protected routes exist after login, but the codebase does not currently enforce them. `README.md` tells users to send `Authorization: Bearer ...` for “próximas rotas protegidas”, yet the controller layer does not declare any protected routes. That mismatch will confuse integrators and hides the missing authorization layer.

## Reliability / Maintainability
- CORS is fully open. `Civitas.WebAPI/Program.cs` calls `AllowAnyOrigin()`, `AllowAnyMethod()`, and `AllowAnyHeader()`. For a browser-facing admin app this is permissive enough to expose the API to any website origin; if the API is ever meant to be private, this should be narrowed per environment.
- The controller layout is inconsistent. The project has both `Civitas.WebAPI/Controllers/` and `Civitas.WebAPI/controllers/` directories. On case-sensitive filesystems and in code navigation tools, this creates avoidable friction and makes the codebase harder to audit.
- `Civitas.WebAPI/Controllers/DespesaController.cs` trusts `IdUsuario` from the request body, and `Civitas.WebAPI/Services/Entities/DespesaService.cs` validates that foreign key directly against the database. If the intent is to bind the creator to the authenticated user, that binding is missing; if not, the API allows arbitrary attribution of expense ownership.

## Test / Coverage Gaps
- The current test suite covers login, pagination, and user validation, but it does not cover authorization enforcement because no route is actually authorized. `Civitas.WebAPI.Tests/AuthEndpointsTests.cs`, `Civitas.WebAPI.Tests/PaginationEndpointsTests.cs`, and `Civitas.WebAPI.Tests/UsuarioValidationEndpointsTests.cs` validate happy-path behavior, but there is no negative test proving that unauthorized requests are rejected.
- There is no test proving startup fails fast when JWT settings or the database connection string are missing. Because `Civitas.WebAPI/Program.cs` reads `JwtOptions` directly and immediately builds a signing key, configuration drift will surface at runtime rather than during startup validation.
